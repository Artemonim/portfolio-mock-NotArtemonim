# **Проект: NotArtemonim — управляемый LLM-участник Telegram-групп**

 [🇷🇺 Русский](README.md) | [🇬🇧 English](README.en.md)

## **Краткий обзор**

`NotArtemonim` — это асинхронная Python-система, в которой языковая модель выступает полноправным участником и модератором группового Telegram-чата. Бот активно работает в 20+ группах, снимая нагрузку с людей-администраторов: накапливает отдельный контекст по каждому чату, принимает решения пакетно, отвечает, молчит или применяет модераторское действие — и даёт оператору понятные рычаги контроля.

![](/media/Router%20Reasoning.png)

Проект делает ставку на product/runtime-зрелость: локальный и облачный inference, многослойную память, объяснимость решений, feedback loop, операторский UX в Telegram и браузере, строгую локальную CI и удобный жизненный цикл процесса. При этом публичное описание намеренно не раскрывает внутренние промпты, точные защитные эвристики и низкоуровневые детали, которые могли бы упростить abuse, взлом или реверс-инжиниринг.

*Код проекта является закрытым.*

*Github Langbar симулирован при помощи [gh-lang-mock](https://github.com/Artemonim/gh-lang-mock).*

## **Технологический стек**

*   **Язык**: Python 3.13+
*   **Telegram runtime**: aiogram 3.26.x, asyncio, long polling
*   **Веб-слой**: aiohttp 3.13.x (локальный WebUI)
*   **Данные**: PostgreSQL, asyncpg, Alembic, async SQLAlchemy 2.0
*   **Конфигурация**: pydantic 2.12.x + pydantic-settings 2.13.x
*   **HTTP и LLM-интеграции**: httpx, LM Studio, OpenRouter
*   **Инструментальный шлюз**: MCP (Model Context Protocol)
*   **Качество кода**: Ruff, MyPy strict, Bandit, pip-audit
*   **Тестирование**: pytest 9.x, pytest-cov, 1200+ test cases with 75+% coverage
*   **Локальная CI**: трёхуровневый конвейер на базе [Agent Enforcer 2](https://github.com/Artemonim/AgentEnforcer2)

## **Что в проекте уже реализовано**

### **1. Поведение в чате**

*   **LLM как участник диалога:** система умеет слушать обычный групповой разговор, а явное обращение к боту лишь повышает приоритет, но не является единственным триггером.
*   **Closed-batch routing:** входящие сообщения сначала собираются в пачки, а уже затем проходят через Роутер. Это даёт более естественное участие в беседе и уменьшает количество импульсивных ответов.
*   **Политика "ответить или промолчать":** модель не обязана реагировать всегда. Система умеет выбирать молчание, ответ, косвенные реакции, вызов субагентов и другие сценарии.
*   **Изоляция контекста по чатам:** память и состояние не смешиваются между разными группами.
*   **Встроенная модерация:** система отслеживает нарушения, применяет лестницу предупреждений и исполняет Telegram-действия — warn, mute, ban, delete — в зависимости от решения Роутера. Каждое действие логируется и остаётся прозрачным для операторов.
*   **Мультимодальность:** бот читает чат, смотрит гифки и картинки; слушает голосовые и видео через [AskVLM]().

### **2. Память, знания и контекст**

*   **OpenClaw-like многослойная память:** активное контекстное окно, рабочие заметки, сжатая per-chat память, глобальные знания и кураторская база знаний работают как разные слои с разной ролью.
*   **PostgreSQL как операционный источник истины:** история батчей, решений, сигналов качества, настроек чатов и review-контур живут в БД, а сжатая память пересобирается из этих данных.
*   **Периодическая компактификация памяти:** отдельный контур уплотняет контекст и удерживает его полезным для следующих циклов.
*   **Кураторская база знаний:** ответы основаны на курируемой базе знаний с операторским контролем.
*   **Предложения обновлений базы знаний:** на основе анализа чатов, субагент Библиотекарь может предложить дополнения к постоянной памяти.
*   **Семантические подсказки:** Админы и пользователи могут оценивать каждый ответ чатбота, а система строит на основе этого систему подсказок с положительными и негативными примерами для каждого отдельного ответа.

### **3. Операторский UX**

*   **Привилегированный DM-канал:** основная операторская поверхность — в личке доступны команды управления и свободный диалог с глобальным контекстом продукта.
*   **Chat registry / onboarding wizard:** в операторском контуре есть пошаговый сценарий настройки чата, включая контекстные параметры вроде часового пояса, чтобы ответы лучше совпадали с реальной средой чата.
*   **CLI для оператора и разработчика:** health checks, пересборка памяти, статистика feedback, управление доступом и другие эксплуатационные операции доступны без GUI.
*   **Глобальный и пер-чатовый mute:** есть операторские kill switch'и для исходящего поведения, не ломающие сам ingest и аудит.
*   **Напоминания и сервисные фоновые сценарии:** проект умеет не только реагировать на сообщения, но и поддерживать вспомогательные workflow'ы вокруг чатов.
*   **Обновление runtime-настроек без полного рестарта:** изменения конфигурации и базы знаний могут обновляться без перезапуска.
*   **Graceful shutdown/drain:** процесс умеет корректно завершаться, не обрывая активные сценарии в случайной точке.

### **4. Надёжность, наблюдаемость и безопасность**

*   **Разделение ingest и inference:** входящий Telegram-поток не должен зависеть от длительности LLM-вызовов.
*   **Отдельные очереди и троттлинг исходящих действий:** контроль частоты сообщений и обработка ограничений Telegram встроены в runtime.
*   **Структурированная наблюдаемость:** есть корреляция между update, batch, decision case и ответом, плюс отдельные runtime-артефакты, логи и диагностические следы.
*   **LLM JSONL-логирование и session artifacts:** удобно для разборов, регрессий и ручной диагностики.
*   **Автоматическая доставка high-signal ошибок оператору:** важные сбои не остаются только в консоли.
*   **Feedback loop по реакциям и голосам:** качество ответов оценивается не только вручную, но и через пассивные продуктовые сигналы.
*   **RBAC / ACL:** минимальная продуктовая матрица ролей уже работает и отделяет глобального оператора от администратора конкретных чатов.
*   **Явное разделение trusted и untrusted контента:** система специально строится так, чтобы пользовательский ввод, tool output и внешний контент не превращались в привилегированные инструкции.
*   **Защитные проверки перед side effects:** действия проходят через контролируемый контур валидации, журналирования и политик.
*   **Local-first админ-поверхность:** чувствительные операторские сценарии ориентированы на локальный контур и контролируемый доступ.

### **5. Инженерная зрелость**

*   **Multi-provider LLM routing:** один и тот же продуктовый сценарий может работать как на локальном inference через LM Studio, так и на облачном маршруте через OpenRouter.
*   **100+ тестовых файлов:** покрыты core/runtime, router, schema, telegram flow, knowledge base, webapp, operator UX и вспомогательные сервисы.
*   **Строгая типизация и локальная quality gate:** mypy strict, ruff, bandit, pip-audit и coverage monitoring встроены в повседневный цикл разработки.
*   **3-уровневая локальная CI:** полноценный локальный конвейер с отчётами, кэшем и профилями запуска.
*   **Миграции и эволюция схемы БД:** 15+ Alembic-миграций, схема активно развивается вместе с продуктом.

## **Что находится в планах**

*   **Расширенный moderation UX:** human-in-the-loop approval для сложных кейсов, детальный audit trail и операторские override-сценарии.
*   **Более широкий агентный цикл:** текущий runtime безопасно работает с инструментами, но multi-step agent loop со всем набором инструментов остаётся в roadmap.
*   **Дополнительные специализированные агенты:** архитектура рассчитана на расширение, часть ролей пока в планах.
*   **Operator analytics и сервисные сценарии:** richer timeline, рестарт-UX, массовые операции и дополнительная диагностика.
*   **Дальнейшее усиление security/eval слоя:** расширения в anti-abuse, red-team и security automation.
*   **Юзербот для обработки тёплых лидов (в разработке):** отдельная подсистема на базе Telegram userbot API (Telethon), которая сможет импортировать список лидов, выходить с ними на контакт и отвечать на вопросы по базе знаний с мягким переводом в управляемый чат.
*   **Telegram Mini App:** опциональная мобильная операторская поверхность как дополнение к DM-каналу.

## **Демонстрируемые навыки**

*   Проектирование управляемого LLM-участника группового чата, а не классического command bot
*   Асинхронный `Python` (`asyncio`, `aiogram`, `aiohttp`, `asyncpg`, `SQLAlchemy 2.0`, `httpx`)
*   OpenClaw-like многослойная память и rebuildable memory artifacts
*   Multi-provider LLM orchestration: `LM Studio` + `OpenRouter`
*   Мультимодальные и media-aware LLM pipeline'ы
*   Интеграция `MCP` как безопасного инструментального слоя для LLM
*   Операторские поверхности: CLI, Telegram DM review, локальный WebUI
*   Автоматическая модерация Telegram-групп: warn, mute, ban, delete с лестницей предупреждений и audit trail
*   RBAC / ACL и админские workflow'ы для Telegram-продукта
*   Feedback loop, explainability и операторский аудит решений
*   Наблюдаемость, graceful lifecycle, hot reload части runtime-настроек
*   Проектирование и эволюция схемы БД через `Alembic`
*   Локальная 3-уровневая CI по философии [Agent Enforcer 2](https://github.com/Artemonim/AgentEnforcer2)
*   Строгая типизация, тестирование и security checks в ежедневном цикле разработки
