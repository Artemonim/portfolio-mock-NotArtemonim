# **Проект: NotArtemonim — LLM-участник группового Telegram-чата**

[In English](./README_ENG.md)

## **Краткий обзор**

`NotArtemonim` — это асинхронная Python-система, в которой языковая модель функционирует как полноправный *участник* группового Telegram-чата, а не как обработчик команд. Сообщения батчатся, после чего **Роутер** вызывает LLM, которая сама решает: участвовать ли в диалоге, ответить ли сейчас, как обновить долгосрочную память. Все решения фиксируются в PostgreSQL; производный слой сжатой памяти хранится в Markdown-файлах и пересобирается фоновым **Библиотекарем**. Для операторов предусмотрены: CLI, встроенный интерфейс проверки в личке, локальный веб-интерфейс и опциональный Telegram Mini App.

*Код проекта является закрытым.*

*Github Langbar симулирован при помощи [gh-lang-mock](https://github.com/Artemonim/gh-lang-mock)*

## **Технологический стек**

*   **Язык**: Python 3.13+
*   **Telegram framework**: aiogram 3.26.x (asyncio, long-polling)
*   **База данных**: PostgreSQL (asyncpg), schema v1+, миграции через Alembic
*   **ORM**: SQLAlchemy 2.0 (async)
*   **Веб-интерфейс администратора**: aiohttp 3.13.x (REST API + статика; Telegram Mini App отложенно)
*   **Конфигурация**: pydantic-settings 2.x (строгая типизация переменных окружения)
*   **HTTP-клиент**: httpx (LM Studio Responses API, OpenRouter)
*   **MCP-интеграция**: mcp 1.2.x (Model Context Protocol — шлюз инструментов для LLM)
*   **LLM-провайдеры**:
    *   LM Studio (локальный инференс, загрузка модели по требованию, мультимодальные модели)
    *   OpenRouter (облачный инференс; список моделей с приоритетом, предпочтения провайдеров)
*   **Инструменты качества кода**:
    *   Ruff (форматирование и линтинг)
    *   MyPy 1.20.x (строгая проверка типов)
    *   Bandit (SAST)
    *   pip-audit (проверка зависимостей на CVE)
    *   [Agent Enforcer 2](https://github.com/Artemonim/AgentEnforcer2) (эталонная архитектура трёхуровневой локальной CI)
*   **Тестирование**: pytest 9.x, pytest-cov (покрытие ≥75%); ~60 тестовых модулей
*   **CI/CD**: 3-уровневая локальная CI (`run.ps1` → `build.ps1` → `build.py`), совместима с Windows PowerShell 5.1+ и PowerShell 7+

## **Ключевые задачи и решения**

*   **«Участник», а не «бот»:** Задача — создать LLM-агента, который живёт в чате как собеседник: накапливает контекст без смешивания между чатами, сам решает, когда вступить в разговор, и помнит договорённости между сессиями. Это принципиально отличается от типичных Telegram-ботов, отвечающих только на явные команды.
*   **Масштабируемый контекст при ограниченном окне LLM:** Полная история переписки не помещается в окно языковой модели. Решение — трёхслойная память: активное окно контекста (ограничено по времени, числу сообщений и размеру батча); слой сжатой Markdown-памяти (краткое изложение и зафиксированные решения), которым управляет Библиотекарь; и PostgreSQL как долгосрочный источник истины, из которого Библиотекарь эту Markdown-память пересобирает.
*   **Контроль качества решений Роутера:** Система фиксирует объяснение каждого решения, а операторы могут проверять пары «батч → ответ» прямо в Telegram (кнопки оценки, «Почему?», отключение пары). Это создаёт замкнутый цикл обратной связи с полным журналом аудита.
*   **Защита от подмены инструкций:** Система явно разделяет доверенный и недоверенный контент в промпте. Провенанс каждого фрагмента отслеживается; попытки внедрить инструкции через пользовательские сообщения обнаруживаются до отправки в модель.

### **Архитектурные и технические особенности**

1.  **Роутер с замкнутыми батчами:**
    *   Входящие сообщения буферизуются Планировщиком батчей, затем Роутер вызывает LLM с собранным контекстом.
    *   Роутер принимает решение по нескольким осям: участвовать ли сейчас, что ответить, что сохранить в память, какие инструменты вызвать.
    *   Для групп поддерживаются два режима: фоновое прослушивание и явное обращение к боту — оба проходят через один конвейер решений.
    *   Роутер реализован как цепочка стадий: политика → список кандидатов → вызов LLM → постобработка → сохранение → выполнение инструментов.
    *   Предусмотрен режим фильтрации рассуждений модели: в Telegram отправляется только итоговый ответ без внутреннего reasoning-блока.

2.  **Трёхслойная память:**
    *   **Активное окно контекста** — формируется динамически при каждом батче: ограничено по давности сообщений, их числу и размеру самого батча. Это «рабочая» память Роутера для текущего решения.
    *   **Markdown-слой** — производный, изолированный: Библиотекарь пересобирает краткое изложение и зафиксированные решения исключительно из строк PostgreSQL, не обращаясь к сырому архиву. Разрешённые пути записи контролируются Менеджером памяти.
    *   **PostgreSQL** — единственный источник истины: хранит все пары «батч → ответ», кейсы решений, оценки операторов и настройки чатов.
    *   **Кэш горячего состояния:** Markdown-сводка кэшируется по цифровому отпечатку файла, что исключает повторное чтение диска между батчами одного чата при неизменном файле.
    *   **Периодический Библиотекарь:** фоновая задача уплотнения Markdown-памяти из PostgreSQL с настраиваемыми лимитами по числу пар, кейсов решений и объёму текста.

3.  **Шлюз инструментов MCP:**
    *   Через адаптер MCP языковая модель получает доступ к инструментам (чтение и запись базы знаний, поиск, предложения обновлений) в рамках формального контракта.
    *   Планировщик агента и его возможности управляют набором доступных инструментов и расписанием выполнения действий.
    *   После каждого вызова инструмента выполняется проверка согласованности состояния чата.

4.  **База знаний с механизмом предложений:**
    *   Кураторская база знаний с разрешительным списком путей для чтения и записи.
    *   Рабочий процесс предложений: Система предлагает обновления базы знаний, оператор — подтверждает.
    *   Проходы и расписание Библиотекаря управляют циклами пересборки.

5.  **Мультимодальные входы:**
    *   При включённой поддержке мультимодальных входов медиасообщения обрабатываются через выборку ключевых кадров и формирование мультимодального промпта для поддерживающих это моделей.
    *   LLM-сервис поддерживает как текстовые, так и мультимодальные маршруты через единый интерфейс.
    *   Для транскрибации голосовых и видеосообщений предусмотрена опциональная интеграция с AskVLM — инструментом собственной разработки на базе Whisper (closed source).

6.  **Многоуровневая операторская поверхность:**
    *   **CLI-команды**: проверка доступности зависимостей, пересборка памяти, агрегированная статистика оценок, управление голосами операторов, управление ролями (назначить/отозвать/просмотреть).
    *   **Проверка в личке**: команды `/review` и `/reviewstats` после приёма сообщений; встроенные кнопки для оценки, просмотра объяснения решения, отключения пары, пагинации.
    *   **Локальный веб-интерфейс** (aiohttp): REST API + статика с аутентификацией по токену; просмотр пар решений, базы знаний, предложений.
    *   **Telegram Mini App**: опциональный HTTPS-эндпоинт с валидацией данных инициализации.

7.  **Наблюдаемость и отказоустойчивость:**
    *   Каждая сессия ведёт лог с манифестом и записями действий.
    *   JSONL-лог всех вызовов LLM с метриками (токены, задержка).
    *   Человекочитаемый транскрипт диалога выводится в консоль.
    *   Необработанные ошибки автоматически доставляются операторам в Telegram.
    *   Все runtime-ошибки сохраняются с контекстом.
    *   Агрегированная статистика и метрики собираются отдельным модулем.
    *   Команда глобального отключения исходящих подавляет видимые действия в Telegram, не останавливая приём и батчинг сообщений.

8.  **RBAC и ACL:**
    *   Продуктовые роли MVP: **product-admin** (глобальный доступ) и **chat-admin** (доступ только к назначенным чатам на уровне строк таблицы).
    *   Служба контроля доступа проверяет права при голосовании, проверке решений и управлении базой знаний.
    *   Отдельный шлюз управляет доступом к DM-интерфейсу.

9.  **Защита от инъекций и безопасность:**
    *   Система обнаруживает попытки внедрения инструкций через пользовательский контент.
    *   Провенанс каждого недоверенного фрагмента в промпте отслеживается отдельным компонентом.
    *   Доверенный контент системной политики и недоверенный пользовательский контент физически разделены при сборке промпта.

10. **3-уровневая локальная CI:**
    *   CI построена по философии [Agent Enforcer 2](https://github.com/Artemonim/AgentEnforcer2) — эталонной архитектуры для надёжных, независимых от языка локальных CI-систем.
    *   `run.ps1` — тонкая обёртка с флагами (`-Fast`, `-SkipLaunch`) и справкой.
    *   `build.ps1` — оркестратор стадий, кэша, логов и итогового JSON-отчёта.
    *   `build.py` — Python-логика стадий: `self-check`, `fmt`, `lint`, `line-limits`, `compile`, `test`, `coverage`, `security`.
    *   Профиль `-Fast` для быстрых итераций пропускает `coverage` и `security`; полный профиль — безопасный дефолт перед передачей результата.

## **Демонстрируемые навыки**

*   Проектирование LLM-агента с Роутером на основе батчей и детерминированным резервным слоем
*   Асинхронный `Python` (`asyncio`, `aiogram 3.x`, `aiohttp`, `asyncpg`, `SQLAlchemy 2.0 async`, `httpx`)
*   Трёхслойная персистентная память: `PostgreSQL` (источник истины) + изолированный Markdown-слой + активное контекстное окно
*   Интеграция `MCP` (Model Context Protocol) как шлюза инструментов для LLM
*   Сборка промпта с явным разделением доверенного/недоверенного контента и защитой от инъекций
*   Мультимодальные LLM-пайплайны (обработка изображений, выборка ключевых кадров, интеграция с транскрибатором)
*   Multi-provider LLM routing: `LM Studio` + `OpenRouter` со списком резервных моделей
*   `RBAC`: product-admin / chat-admin с ACL на уровне строк в `PostgreSQL`
*   Веб-интерфейс администратора (`aiohttp` REST + статика + Telegram Mini App)
*   Встроенный интерфейс проверки решений в Telegram DM с журналом аудита и кнопкой объяснения
*   Наблюдаемость: JSONL-логирование LLM, сессионные манифесты, автоматические уведомления об ошибках
*   Проектирование и тестирование схемы БД (`Alembic`, schema v1 + миграции)
*   ~60 тестовых модулей: unit + интеграционные тесты всех подсистем
*   3-уровневая локальная CI по [Agent Enforcer 2](https://github.com/Artemonim/AgentEnforcer2) с кэшем, инкрементальными проверками и JSON-отчётом
*   Операторский CLI с полным набором команд и кодами завершения
